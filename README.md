# TermRepoWebSite

`TermRepoWebSite` 是 TermRepo 项目的网页前端工作区，当前包含两个彼此分工明确的 Vue 站点：

- `termrepo-guide-site`
  面向用户的指南站，负责介绍产品定位、核心功能、使用方式、常见问题与路线信息。
- `termrepo-control-site`
  面向增强能力接入的管理站，负责 token、内测资格、AI 能力、云同步、工单等服务入口与状态管理。

它们共同服务于同一个产品体系，但职责不同：

- 指南站负责说明与引导
- 管理站负责接入与管理

## 项目定位

TermRepo 的核心不是做一个重 SaaS 官网页面群，而是围绕“开发者术语积累、理解、管理与复用”构建清晰、克制、工具感强的前端入口。

这个仓库中的两个站点都应遵守相同的产品边界：

- 本地优先
- 轻接入
- 不依赖重登录作为基础前提
- 云能力属于增强层，而不是本地能力的替代品

## 仓库结构

```text
TermRepoWebSite/
├─ termrepo-guide-site/      # 指南站前端
├─ termrepo-control-site/    # 管理站前端
├─ LICENSE
└─ README.md
```

两个子项目当前都基于以下技术栈：

- Vue 3
- TypeScript
- Vite
- Vue Router
- Pinia
- Axios

## 两个站点分别负责什么

### `termrepo-guide-site`

适合承载的内容包括：

- 首页
- 功能介绍
- 使用指南
- 常见问题
- 更新日志
- 项目定位与路线说明

不应承担的职责包括：

- token 管理
- 资格发放
- 用户服务配置
- 工单处理
- 重型后台逻辑

### `termrepo-control-site`

适合承载的内容包括：

- token 获取与管理
- 内测资格与权益说明
- AI 功能开关与额度状态
- 云同步状态与设备信息
- 工单与反馈入口
- 插件接入说明与诊断信息

它的定位是“增强能力接入站”，而不是传统意义上的重后台系统。

## 环境要求

- Node.js: `^20.19.0 || >=22.12.0`
- npm

## 开发方式

两个子项目相互独立运行、独立安装依赖。

### 启动指南站

```bash
cd termrepo-guide-site
npm install
npm run dev
```

### 启动管理站

```bash
cd termrepo-control-site
npm install
npm run dev
```

### 常用命令

两个子项目都支持相同的命令：

```bash
npm run dev
npm run build
npm run preview
npm run type-check
```

## 当前状态

### 指南站

指南站已经从默认脚手架页面演进为真实项目页面，当前重点是：

- 对外表达 TermRepo 的定位
- 说明插件已具备的本地闭环能力
- 以中文内容为主组织项目介绍与使用路径
- 维持接近 Vercel 风格的克制视觉系统

### 管理站

管理站当前是控制台前端骨架与业务界面实验场，目标是逐步承接：

- token 接入
- AI 服务状态
- 云同步能力
- 资格管理
- 工单与反馈

如果当前后端尚未完全就绪，管理站可以继续以 mock 或前端模拟数据方式推进页面与交互设计。

## 文档边界建议

在维护这个仓库时，建议继续保持下面这条分工原则：

- 用户想了解“TermRepo 是什么、怎么用”时，看指南站
- 用户想“开通能力、配置服务、查看状态”时，用管理站

不要把两个站写成同一种网站，否则会很快出现职责混乱：

- 指南站会变得像后台
- 管理站会变得像营销页

## 一句话总结

`TermRepoWebSite` 现在不是单一指南站仓库，而是同时承载“指南站 + 管理站”两个前端入口的网页工作区。

## termrepo-admin-site

`termrepo-admin-site` 是独立管理站，面向服务部署者/管理员。它通过 `VITE_API_BASE_URL` 连接后端，用于配置全局 AI 模型、默认额度和用户权限。普通用户仍使用 `termrepo-control-site`。
